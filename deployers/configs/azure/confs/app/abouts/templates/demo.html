展示需求說明

資料應用的議題中，命題是最重要的關鍵！

請參考下圖，一個跨國企業在全球十個國家，四十一個商業據點，銷售多種消費電子產品，每個據點除了以提供實體門市之外，也佈署不同業務性質的網站系統與銷售系統，服務在地消費者，企業總部主要掌握各類產品的開發，也掌握消費者需求的變化趨勢

現在，身為企業總部資訊主管的你，被要求在不影響現有資訊系統運作的前提下，將所有營業據點中大量的客戶活動資訊以及交易資訊，集中蒐集後，提供以下數據應用︰

應用於即時統計

  • 企業產品與服務在全球用戶視為最熱門（點擊率最高）、最乏人問津（點擊率最低）、意見及問題反映最多的各項實時排名
  • 依產品、國家、據點等，實時統計出各業務通路據點產生的銷售、客訴、退貨的交易紀錄，做成交叉比對報告，提供營業主管線上實時查詢

應用於分析商情

  • 找出各類商品在全球的網站活動與交易的關聯性
  • 預測全球網站活動與商品交易的趨勢

首先，我們以圖解方式表達上述的功能要求：

上圖中，有幾個角色出現在企業總部的需求中︰

  • Application Servers︰座落於全球各地的資訊系統主機，資料的提供者
  • Data Collector︰將座落於總部，負責收攏各地送來的資料
  • Data Processor︰將座落於總部，負責將資料儲存、統計、歸納、分析
  • ODS︰將座落於總部，保存原生數據，以及經過處理的數據
  • Reporter︰將座落於總部，產生適當的圖表與數據，提供業務主管以及企業決策層使用

接著，我們將資料被蒐集、被處理、直到被提出附加應用的作業過程，畫出細部流程：

資料需要在上述流程的每一個步驟中被有效地處置，才能滿足企業總部所提出的需求，我們簡單地描繪每個步驟的功能需求︰

收集︰收集的對象是網站活動與交易紀錄，這裡不是抽樣調查，而是完整數據。

佇列︰主要處理高並發，保證訪問順利，這就需要大量的資料庫支撐，依靠合理的分流、公有雲等架構方法，保證每一個數據的準確有用。

清洗︰採集好數據，肯定不少是重複或是無用的數據，此時需要對數據進行處理，去掉雜訊和無關數據

落地︰將這些來自前端的數據導入到集中的大型分布式資料庫，或者分布式存儲集群

分析︰這裡是指資料探勘的過程，為了能找出適當的演算法與數據參數

建模︰使用選定的演算法處理數據，計算分析然後提取出模型

處理︰數據在這裡被使用，例如統計以及預測

快取︰存放即時統計的結果，提供給應用系統使用

應用︰以圖表的形式，提供資料統計、分析、歸納、關聯以及預測的結果

架構規劃說明

上述要求中有幾個主要的設計目標：

  • 需要進行分布式資料處理。
  • 需要實時數據處理
  • 需要高吞吐率
  • 需要水平擴展

因此，我們示範以 Kakfa 做為各網站活動流以及交易紀錄接收平台，以 MongoDB 提供資料落地，以 apache Storm 進行分類與統計的作業，再將統計結果實時傳遞給線上實時資料庫 Redis Cluster，供網站報表系統取用，參考以下系統規劃圖：

這樣的規劃具有以下幾點好處：

  • 異質路網、異質系統快速整合，將既有系統客製化影響降至最低。
  • 可逐步加入離線資訊處理的應用。
  • 不必為交易紀錄或活動流數據量尖峰擔心負載過量而造成平台崩潰。
  • 不用擔心數據被處理前遺失。

展示架構說明

我們將上述系統規劃稍作簡化，嘗試在展示環境中模擬應用案例的系統架構：

我們以 python 程序簡化並模擬在應用情境中的幾個任務：

  • 應用系統群組向 Kafka 發送活動流數據以及交易資料。
  • Apache Storm 叢集自 Kafka 消費所有訊息，進行分類、統計後，寫入快速查閱資料庫。
  • Data Analyze Cluster 中. 週期性資料分析作業，包括活動與交易資料關聯分析、聚類以及產出預測模型。
  • 報表主機自快速查閱資料庫查詢實時統計數據，以及使用預測模型產出預測數據，提供報表功能。
